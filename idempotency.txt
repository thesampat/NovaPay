# NovaPay: Idempotency & Distributed Integrity Design

## 1. Key Generation (The Fingerprint)
- **Problem**: Arithmetic addition (`receiver_id + request_id`) causes collisions (e.g., `100+1 == 101` and `101+0 == 101`).
- **Solution**: Use **String Concatenation** (Composite Keys).
- **Format**: `sender_id:receiver_id:batch_id`
- **Rule**: Generation must be **Deterministic**. Re-running the same batch must always produce the exact same child IDs.

## 2. Distributed Architecture (The Layers)
- **Layer 1: The Shield (Redis)**
    - Fast, in-memory check to reject spammy retries instantly.
    - Reduces load on the primary database.
- **Layer 2: The Vault (Database - MongoDB/SQL)**
    - The final source of truth for financial integrity.
    - Handles "System Crash" scenarios where Redis might be out of sync.

## 3. Storage Strategy (The "FAANG" Way)
- **Embedded Array (Option A)**:
    - Save IDs directly in `User.processed_transactions: string[]`.
    - Allows for a **Single Atomic Update** (`updateOne` with `$push` and `$ne`).
    - Best for simplicity and raw performance.
- **Separate Collection (Option B - Industry Standard)**:
    - Dedicated `IdempotencyKeys` collection.
    - **TTL Index**: Automatic cleanup after 24-48 hours (`expireAfterSeconds`).
    - Requires **Database Transactions** (Mongoose Sessions) to stay in sync.

## 4. The Crash Scenario (Survival)
- **The Risk**: System crashes AFTER money is deducted but BEFORE the result is cached in Redis.
- **The Fix**: The Database must check-and-set the request_id **at the same time** it updates the balance. This ensures that if the money moved, the "receipt" is already saved on disk.

## 5. Metrics & Resume Bullets
- **Throughput**: "Handled 3000+ RPS using layered idempotency."
- **Reliability**: "Achieved 100% financial consistency across 10k concurrent transactions."
- **Efficiency**: "Reduced DB load by 95% using Redis as an idempotency shield."
- **TTL**: Standard window is **24-48 hours** for safety and storage efficiency.
